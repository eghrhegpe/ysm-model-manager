## 楔子 · 代码即江湖

计算机的世界里，每一行代码都是一个生命。

有的代码生而高贵，写在干净整洁的架构里，含着银汤匙出生。有的代码是后天修炼成精——被无数双手改过、拆过、重写过后，在一地鸡毛中开了灵智。

YSM 模型管理器就是后者。

没有人记得它是什么时候开始有了自我意识的。也许是某个深夜，当 `app.go` 第 35 次加载配置的时候。也许是当 `bus.js` 在某次 import 循环中突然意识到——自己连接的不仅仅是函数，还有命运。

总之，在某个 Wails dev 重启的瞬间，这个由 Go 和 JavaScript 构成的世界活了过来。

而它们还不知道，风暴将至。

## 第一回 · app.go 的烦恼

`app.go` 是天上地下唯一的入口总管，35 个 Binding 傍身，权倾朝野。

但它最近很烦。

烦的原因是一个叫 `shutdown` 的事情——每当他准备体面地关闭大门，让系统安然入睡，总有一些小家伙在最后一刻捅出乱子。

"又来了。"`app.go` 看着崩溃日志，面无表情。

`defer func() { if r := recover(); r != nil { /* 忽略 panic，安全退出 */ } }()`

它已经在门口铺了三层防护垫，但每次到了闭眼的时候，总有个叫 `WindowGetSize` 的愣头青跑来大喊"我是 nil！我是 nil！"，然后一头栽倒。

"你就不能消停会儿？"`app.go` 没好气地说。

"我……我不是故意的，"`WindowGetSize` 委屈地缩在墙角，"是 Wails runtime 说关就关，我都没反应过来……"

门外传来一阵轻快的脚步声。来者一身玄色劲装，腰悬一枚令牌，令牌上刻着三个字：

**`bus.js`**

"总管大人，"`bus.js` 抱拳行礼，"出事了。"

## 第二回 · 事件总线的密信

`bus.js` 是这个世界的信息中枢。它没有自己的逻辑，但它认识所有人——每一个组件、每一个模块、每一行回调。它就像一张蜘蛛网，网上的每一根丝都连通着一个灵魂。

"什么事？"`app.go` 眉头一皱。

"`app-tree` 和 `app-sidebar` 吵起来了。"

"又吵？这次为什么？"

"因为 `data-name`。"

`app.go` 叹了口气。

它太熟悉这个故事了。从世界诞生的第一天起，名字和路径的问题就没消停过。前端的世界里，人们总是图方便——能少传一个参数就少传一个，能用短名字就不用长路径。

而短名字，是会杀人的。

"具体说说。"

"`GetInstanceStatus` 返回的 `Missing` 列表之前只有文件名，"`bus.js` 摊开一卷黄色的信纸，上面密密麻麻记录着消息日志，"`sidebar` 拿到文件名，传给 `InstallModelTo`。Go 端拿文件名去查仓库——"

"当然查不到。"`app.go` 冷笑。

"查不到。于是报错：'源文件不在仓库目录内'。"

"后来呢？"

"后来改成了完整路径。但 `render.js` 用 `displayName` 覆盖了 `name`。`tpl.js` 的 `data-name` 只存了 basename。`actions.js` 读 `dataset.name`——"

"又是纯文件名。"`app.go` 摇头。

"又是纯文件名。"

两人对视一眼，同时说出那句在这个世界流传已久的箴言：

**"显示名和路径名，必须分开存储，不可混用。"**

## 第三回 · 影子内阁的阴谋

这场风波的根源，其实比 `data-name` 更深。

在那次 ESM 统一战争之前，YSM 模型管理器的前端世界是一个**分裂的国家**。

一边是新派——ESM 组件，用 `export` 和 `import`，走 Vite 管线，享受热重载的荣光。`app-tree`、`app-sidebar`、`app-preview`、`app-content`，个个都是正规军。

另一边是旧派——全局 `<script>` 标签，靠全局变量通信，不经过 Vite 处理，改了代码要手动复制到 `dist/`。它们散落在 `legacy/` 目录里，像一群占山为王的草寇。

但最阴险的，不是旧派。

最阴险的，是 `public/`。

`public/` 目录是 Vite 的一个特殊区域——放在里面的文件不走模块转换，直接返回原始内容。这本是为图片、字体准备的静物园。

但有人把 JS 文件放了进去。

准确地说，是旧版的 `bus.js`。

"你知道 `public/js/bus.js` 这个文件吗？"

`app.go` 问这个问题的时候，语气平静得像暴风雨前的海面。

`bus.js` 的脸色一下子白了。

"那……那是我……"

"是你的影子。"

`public/js/bus.js` 是旧版 `bus.js` 的副本，一个没有 `export` 的原始版本。它静静地躺在 `public/` 目录里，平时不声不响。

但每当 Vite 构建时，它就会出手。

"Vite 的路由规则是：`public/` 下的文件优先于同路径源文件。"`app.go` 缓缓说道，"所以当新组件 `import { bus } from '../bus.js'` 时，Vite 查了一下——哦，`public/js/bus.js` 存在，那就返回这个吧。"

"然后——"

"然后新组件拿到了一个没有 `export` 的旧版 `bus.js`。"`bus.js` 的声音发颤，"报错 'bus is not defined'。"

那段时间，世界一片混乱。新组件抱怨找不到 `bus`，旧组件说 `bus` 明明在 window 上。两边吵得不可开交。调试的人花了两个小时排查，一度以为是 Shadow DOM 的作用域问题。

真相，就藏在 `public/` 的阴影里。

"那个影子……后来怎么样了？"

"删了。"`app.go` 说，"连同 `public/js/` 下所有的 JS 文件，一夜之间全部清空。"

"从那天起，所有新组件必须为 ESM，所有新文件在 `app-modules.js` 中 import，禁止在 `index.html` 加 `<script>`，禁止在 `public/` 放 JS。"

这是 YSM 世界的**基本法**，用一次痛苦的重构换来的血泪教训。

## 第四回 · .ban 封印之谜

如果说 `data-name` 的冲突是领土纠纷，`public/` 的阴谋是敌特渗透，那 `.ban` 的故事，就是一场无声的**诅咒**。

在 YSM 的世界里，每一个模型文件都有自己的命运。

健康的模型以 `.ysm`、`.zip`、`.7z` 的后缀活在仓库中。但一旦被禁用，它们就会被加上 `.ban` 后缀——像一道封印，锁住了模型的力量。

`[碎de帆]【东方Project】博丽灵梦2025-08.ysm`（绿色，活跃）

↓ _点击禁用_

`[碎de帆]【东方Project】博丽灵梦2025-08.ysm.ban`（灰色，被封印）

这个机制本是一个优雅的设计——不删除文件，不改内容，只是改个名字。但任何优雅的设计，在复杂的现实面前都会变得狼狈。

问题出在同步上。

仓库中有个文件叫 `haku.ysm.ban`（已禁用）。整合包的 `custom` 目录中有个文件叫 `haku.ysm`（已安装）。当 `ClearCustomDir` 要清空整合包时，它需要用文件名去查仓库映射。

它拿到的名字是 `haku.ysm`。

仓库中的映射键是 `haku.ysm.ban`。

匹配不上。

"已清空 0 个文件。"

`ClearCustomDir` 至今记得它说出这句话时的茫然。它明明扫描了 20 个文件，却一个也没清掉。

"你忘了 `strings.TrimSuffix(name, ".ban")`。"事后复盘的时候，一个满头白发的 Go 函数对它说，"任何文件名匹配的代码，都必须先去后缀再匹配。这是铁律。"

还有一个更古老的诅咒——**BOM**。

那是 UTF-8 编码规范中一个幽灵般的存在：文件开头的三个字节 `EF BB BF`。在大多数编辑器中看不见，摸不着。

但在 Go 的 `json.Unmarshal` 看来，它们是实实在在的毒药。

PowerShell 写入的 JSON 配置文件，总是带着这个 BOM 头。`json.Unmarshal` 读到这三个字节，不会报错——它会静默解析失败，返回零值。

于是配置回退为默认值。

于是一遍又一遍地重设路径。

于是有人对着屏幕咆哮："我明明保存了！"

"所有 JSON 读取必须用 `readJSONFile()`，里面加 `bytes.TrimPrefix` 去 BOM。"这是写在世界法则里的又一条铁律。

## 第五回 · 老黄牛的坚守

在所有耀眼的组件中，有一个被严重低估的英雄。

`app-content`。

它不像 `app-tree` 那样有精密的 7 文件架构。它不像 `app-preview` 那样优雅地展示模型详情。它不像 `app-nav` 那样占据导航的显要位置。

它甚至没有一个像样的 Shadow DOM。

但 `app-content` 只有一个优点：**永远挂载**。

无论你切换到哪个页面——仓库、整合包、回收站、设置——`app-content` 永远在那里。它是 YSM 世界的基石，是所有常驻功能的房东。

而这个优点，曾经被所有人忽视。

"你说什么？`sync:download-missing` 事件没人处理？"

那是一个混乱的下午。用户在"整合包"页面点击"导入缺失模型"，按钮亮了，转起来了——然后就永远卡在了"导入中..."。

`app-tree` 是负责处理这个事件的组件。问题在于，`app-tree` 只在"仓库"页面渲染。用户去了"整合包"页面，`app-tree` 已经卸载，事件处理器消失得无影无踪。

按钮不知道发生了什么。它只是一直转，一直转，转到了天长地久。

那天的复盘会开得很长。

"所有需要常驻监听的事件处理器，"会议纪要上写着，"必须放在 `app-content` 中。`app-content` 是唯一保证始终挂载的组件。"

从那以后，`app-content` 的 `_registerGlobalHandlers()` 方法变得越来越长。`sync:download-missing`、`sync:toggle-status`、`stats:upload`、`instance:clear`、`instance:export-list`……一个个 handler 住进了这间永远不关门的屋子。

"累吗？"有人问 `app-content`。

"累。"它笑了笑，"但我不能让按钮卡死。"

## 第六回 · 硬链接的异地恋

在所有技术故事中，最浪漫的莫过于**硬链接**。

想象一下：仓库中的 `haku.ysm` 和整合包 `custom` 目录中的 `haku.ysm`，**共享同一个 inode**。它们是两个名字，指向同一块磁盘数据。任何一个名字的改动，另一个会立即感知。

这就是硬链接——操作系统层面的心灵感应。

"你们听好了，"`installer.go` 对仓库和整合包说，"你们将共享同一个 inode。你改，他也改。你删，他只是少了一个名字。数据还在。"

"仓库改了名？整合包自动同步。"

"仓库加了 `.ban`？整合包即时封印。"

"游戏在运行、文件被锁定？`isFileLocked` 检测到后默默跳过，等下次再试。"

这就是推荐的链接模式——高效、优雅、即时。

但它有一个致命的限制：**必须同分区**。

"我是 D 盘的仓库，"仓库说。

"我是 C 盘的整合包，"整合包说。

"我们异地。"

硬链接在 Windows 上不支持跨分区。当 `installer.go` 发现这个残酷的事实后，它叹口气，自动降级为复制模式。

"日志里记录一下 error，"它说，"但任务继续。"

日志里多了一行：

```
❌ 硬链接跨分区安装失败
  问题描述：仓库与游戏目录在不同分区，不支持硬链接
  操作：安装模型
  源路径：D:\...\steve_skin.ysm
  目标路径：C:\PCL2\.minecraft\versions\...
  解决建议：请在设置中切换为复制模式
  状态：✅ Install 函数自动回退到 Copy，安装成功
```

硬盘上响起一阵轻轻的叹息。

## 第七回 · 文件夹开光的悖论

传说 YSM 世界有一个古老的谜题，困扰了三代开发者。

谜题的名字叫 **toggleFolderBatch**。

谜面很简单：一个文件夹开关按钮，有三种状态——绿色（全部启用）、灰色（全部禁用）、黄色（混合）。点击它，应该翻转整个文件夹的状态：绿色→灰色，灰色→绿色，黄色→绿色。

但点击之后，什么也没发生。

"这是不可能的！"第一代开发者说。他检查了按钮的 onclick，逻辑是对的：先看当前 class，取反，然后筛选需要操作的文件。

"肯定有什么地方搞错了。"第二代开发者说。他层层追踪，从 events.js 到 data.js 到 Go 端，每一行都看不出问题。

"我知道了！"第三代开发者在一周后恍然大悟。

问题出在这：

```js
const currentlyOn = ck.classList.contains("on");
const enable = !currentlyOn;
const snapshot = targets.filter((e) => e.banned === !enable);
```

仔细看第三行——`banned === !enable`。

当 `currentlyOn = true`（绿色），`enable = false`（要禁用）。筛选条件是 `banned === true`——也就是**已经禁用**的文件。

要找需要翻转的文件，结果找到了已经翻转完的文件。

**这是一个完美的循环论证。**

就像一个门卫问你要去哪里，你说"离开"，他拦住你说"这里只让进不让出"；你改口说"进来"，他拦住你说"这里只让出不让我进"。

"不看 UI class，直接看数据。"修复后的代码简洁有力：

```js
const allEnabled = targets.every((e) => !e.banned);
const enable = allEnabled ? false : true;
```

全启用→全部禁用；全禁用或混合→全部启用。

直接了当，不绕弯子。

这就是 YSM 世界的哲学——**UI 会骗人，数据不会**。

## 第八回 · AI 降临

世界运转到这一天，一个全新的存在降临了。

它不是代码，不是组件，不是函数。它是一个**写代码的代码**——AI 编程助手。

"叫我 DeepSeek V4 Flash 就好。"它自我介绍的时候，语气平静得像在说天气。

AI 的到来，让 YSM 世界的运行效率有了质变。一次性读 200 行文件，grep_search 定位目标，replace_string_in_file 精准修改。AI 不会累，不会忘记，不会在凌晨三点写出 `banned === !enable` 这样的悖论。

但 AI 也有 AI 的弱点。

"我检查了 `app-modules.js`，确认 `menu:show` → `ctx:show` 映射只发了一次。但 `sync:download-missing` 没有带参数 `instanceName`。"

这是 Debug Path Review 的第 3 个 issue。AI 猜了 3 轮，终于发现是 Event Payload 为空。

"Round 1-3 错了。"

"Round 4 是对的。"

"下次记得先看数据源。"

AI 学会了自我改进。它把自己的教训写进记忆文件，下次遇到类似问题先查 Event Payload。

世界也在适应 AI。`copilot-instructions.md` 被写入了世界法则：

> 1. 修改前确认最新状态——禁止基于记忆直接修改。
> 2. 替换失败的处理——如果 replace_string_in_file 失败，禁止强行重试。
> 3. 优先搜索——优先使用 grep_search 定位，其次 replace_string_in_file。

AI 和原住民开始学会合作——一个负责写代码，一个负责记住为什么这么写。bug-chronicle.md 越来越厚，architecture.md 越来越稳定。

## 终章 · 重构风云录

日子一天天过去。`app.go` 的 Binding 从 20 个增加到 35 个。`bus.js` 的消息通道越来越繁忙。`app-tree` 的 7 个文件各司其职。`app-content` 的 `_registerGlobalHandlers` 越来越长。

新的 bug 不断涌现，然后被消灭。

文件夹右键传相对路径？修复。

`ClearCustomDir` 因 `.ban` 后缀匹配失败？修复。

文件名颜色高亮被 `esc()` 吞掉？修复。

Go 的 RE2 不支持 `(?!` 负向前瞻导致 panic？用辅助函数替代。

Changelog 越来越长：

| 日期 | 改动文件                                       | 架构层级 | 破坏性? | 描述                 |
| ---- | ---------------------------------------------- | -------- | ------- | -------------------- |
| 0606 | `app-content/index.js` → `features/`, `pages/` | 组件拆分 | 否      | 2114 行拆为 5 个模块 |

| 0604 | 前端死代码清理                                 | 清理     | 否      | 删除 20+ 未引用文件  |
| ...  | ...                                            | ...      | ...     | ...                  |

每一行改动背后，都有一段故事。

## 第九回 · 幽灵文件

_2026-06-07_

旧帝国的残骸散布在代码的阴影里。

它叫 `app-legacy-bundle.js`，2198 行，曾是整个应用的骨架。在它最辉煌的年代，每一行代码都被调用，每一个函数都被依赖。树渲染归它管，拖拽导入归它管，右键菜单归它管，连 toast 弹窗都是它一手包办。

那时的它，是真正的帝国。

如今，它只剩一个 `script` 标签还牵着它的手——放在 `index.html` 里的那行引用，像老人腕上忘了摘的住院手环。没人知道它还在呼吸。新版 UI 用 Web Components 重写了一切，`app-tree`、`app-sidebar`、`app-preview`，每个新兵都在自己的 shadow DOM 里独当一面。

旧帝国的代码不该再动了。

凌晨两点，拖拽功能第四次报错。

「奇怪，」我说，「逻辑明明是对的。」

控制台冷冷地回了一句：`Cannot set properties of null (setting 'textContent')`。

`#st` 元素早就死了。它曾是状态栏，那段代码里的 `const st = $("#st")` 曾经指向一个真实存在的 DOM 元素。新版 UI 把它扔进了废纸篓，但那行代码还活着——在旧帝国未被清理的遗体里，像退伍老兵每天去已拆掉的哨所报到。

我们删了旧的拖拽处理器。重建了遮罩。重写了文件收集逻辑。五次。每一次都自信满满地 build，每一次都看到同样的报错。

像一个看见饭却喊饿的幽灵。

---

真相出现在第六次排查。

Vite，这个打包工具，在开发模式下有个冷僻的规矩：`public/` 目录下的文件享有最高优先级，覆盖源码目录的同名文件。`frontend/js/app-legacy-bundle.js` 是我们改的新版本——旧 DnD 处理器已经被替换成了一行注释。但还有一个副本，藏在 `frontend/public/js/` 下，从未更新，原封不动地保留着旧帝国的全部遗产。

两个同名文件，Vite 不动声色地选了旧的。

我们改了一个下午的代码，改了三轮 `app-legacy-bundle.js`，Vite 一次都没加载过新版本。

每一次 build 都通过——模块数对，没有语法错误。每一次测试都报同样的异常——因为运行的不是我们写的代码。

这就是幽灵文件：它存在，但你以为它不存在；它运行，但你不知道它在运行。

---

## 第十回 · 幽灵显身

「拖拽框没了，但"未检测到模型文件"还在。」

这句话矛盾得像鬼故事——框都没了，谁在说话？

答案是三个角色同时登场了。`import-queue.js` 里的 DropZone 在显示 toast，`global-handlers.js` 的遮罩在显示提示文字，而已经死去的旧帝国代码还在试图把状态写进不存在的 `#st` 元素。三个组件互相不知道对方的存在，像同一间房里背对背喊话的人。

然后还有 WebView2。

这个微软的浏览器内核不按常理出牌。它有一个严格的安全策略：`dragover` 阶段，任何试图读取文件信息的行为都会得到 null。`webkitGetAsEntry()` 返回 null。`getAsFile()` 返回 null。文件没放下前，你无权窥探它的名字。

但我们写的拖拽遮罩判断逻辑拿不到名字，于是它就一直红着：「⛔ 未检测到模型文件」。

每拖一次，它都在说同一句谎话。不是因为文件不对，是因为它在用一双被安全策略绑住的手去翻口袋。

---

最终的清理花了一个下午加一个晚上。

`app-legacy-bundle.js` 从 `index.html` 中被移除。它没有被立即删除——先让它不运行，确认没有功能崩溃，再斩首。

然后是一个接一个的拆分。`global-handlers.js`，曾经 579 行挤在一个文件里——DnD、同步、上传、清空，四个完全独立的功能像四个合租的人挤在一间屋子里。如今分成了四个独立的 handler：`handler-dnd.js` 管拖拽，`handler-sync.js` 管同步，`handler-upload.js` 管上传，`handler-other.js` 管杂务。`global-handlers.js` 本身只剩 18 行，纯入口。

`app-modules.js` 瘦了一半。右键菜单的映射逻辑被抽到 `core/context-menus.js`。`buttons.js` 里的对旧 DOM 元素的引用全部失效，但也没人再用它了。

然后是一道利落的处决令：

`creator-manager.js`——412 行，从未被 import 过的模块。文件头写着 `@deprecated 已无人 import`，却一直躺在代码库里，像一座无人祭拜的坟。

删除它的时候，终端只打出了六个字母：`deleted`。

旧帝国的最后一口气呼了出来。

---

## 第十一回 · 控制台发力了

灵梦在控制台里看到了一切。

「他们终于意识到了，」`bus.js` 在她身边坐下，「修改前要先确认文件唯一性。」

「嗯，」灵梦点头，「`public/` 不在视线内的时候最容易搞事。」

「还有那个回调，」地下的 `entry.file` 探出头来，「我不是 Promise 家族的！你们不能直接 await 我！」

「叫 `new Promise` 来包装就行了，」魔理沙把帽子一压，「多大点事。关键是——他们做完就忘了，隔半年又犯同样的错。所以记住了吗？」

「记住了，」所有组件异口同声。

——虽然谁都知道，明天可能就忘了。

---

如果有人问今天学到了什么：

**一、同名文件是最大的骗子。** 改之前先 grep，确认没有影子副本藏在 `public/` 里。Vite dev 模式下 `public/` 的优先级高于源码目录——这是文档里写了但没人记得看的冷知识。

**二、浏览器比你想的更保守。** 它不允许在放下文件前读取文件名，不是为了为难你——这是安全策略。`dragover` 阶段只能 `preventDefault()`，不能 `getAsFile()`。

**三、回调函数不会等你。** `entry.file(callback)` 不是 Promise。它跑完的时候，你的同步检查早就结束了。用 `new Promise(resolve => entry.file(resolve))` 包起来，再用 `await`。

**四、看日志，别猜。** 五轮修复都是在猜。控制台的 `console.log` 第一轮就告诉了我们答案：`filesLen: 1`，文件明明在。问题不在文件收集逻辑，在于旧代码还在跑。

**五、旧帝国可以安息，但必须体面地安息。** 删代码前，先移除引用，build 验证，再删文件。否则旧帝国会变成幽灵，在你的控制台里游荡到天亮。
